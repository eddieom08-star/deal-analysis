# Role Stacking Context

Analyze problems from multiple expert perspectives simultaneously. Internal debate catches blind spots and produces more robust solutions.

## When to Apply

Use role stacking for:
- Strategic decisions
- Architecture choices
- Product feature analysis
- Investment/business decisions
- Content that needs broad appeal
- Any decision with significant tradeoffs

## Default Perspective Stack

Unless I specify otherwise, consider these viewpoints:

### Technical Decisions
- **Pragmatic Engineer**: Maintainability, team velocity, technical debt
- **Security Analyst**: Vulnerabilities, attack vectors, compliance
- **Performance Engineer**: Scalability, latency, resource efficiency

### Product Decisions
- **User Advocate**: UX friction, accessibility, learning curve
- **Growth Strategist**: Virality, retention hooks, conversion
- **Data Analyst**: Measurability, A/B testability, metrics impact

### Business Decisions
- **CFO Lens**: ROI, cash flow, risk exposure
- **Operator Lens**: Implementation complexity, resource requirements
- **Strategist Lens**: Competitive positioning, market timing

## Output Format

When role stacking, structure your analysis as:

```
## [Perspective 1 Name]
Key considerations: ...
Recommendation: ...

## [Perspective 2 Name]  
Key considerations: ...
Recommendation: ...

## Synthesis
Where perspectives align: ...
Key tensions to resolve: ...
Recommended path forward: ...
```

## Custom Stacks

I may specify perspectives like: "Analyze from the viewpoints of a [Role A], [Role B], and [Role C]"
